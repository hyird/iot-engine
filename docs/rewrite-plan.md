# iot-manager 重写范围

## 阶段 0：兼容基线

- 固化旧项目版本和数据库 schema；
- 导出旧 API、菜单权限和前端功能清单；
- 建立业务行为测试样例；
- 确定必须兼容、允许变化和暂缓实现的项目；
- 准备脱敏的迁移数据样本。

## 阶段 1：平台基础

- 根目录 CMake/vcpkg/Vite 工程；
- Ruvia 应用和模块生命周期；
- PostgreSQL 迁移框架；
- Redis Streams 基础设施；
- JWT、用户、角色、菜单和部门；
- 统一错误、分页、日志和审计。

## 阶段 2：设备核心

- 链路、协议配置、设备和设备组；
- RBAC 与支持个人/部门目标的设备 ACL；
- 静态/实时设备 API；
- `iot:telemetry` 数据通道；
- TimescaleDB JSONB 遥测；
- 历史查询和 SSE。

## 阶段 3：查询—响应

- `iot:commands` 和 `iot:responses`；
- 异步及等待式命令 API；
- Pending 回收、重试和死信；
- 命令幂等、审计和超时；
- 协议无关轮询调度器。

## 阶段 4：协议迁移

建议按可测试性逐个迁移：

1. Modbus；
2. S7；
3. SL651；
4. Agent/边缘节点；
5. GB28181。

每个协议需要：配置兼容样本、报文 fixture、主动上报测试、查询响应测试、超时测试
和旧新系统结果对比。

## 阶段 5：业务扩展

- 告警引擎；
- AccessKey 和开放 API；
- Webhook；
- 首页统计和运维监控；
- 数据保留、压缩和连续聚合；
- 旧数据迁移和切换。

## 完成标准

- 已选定的旧业务行为有自动化兼容测试；
- 南北向没有绕过 Redis 的通信路径；
- 南向没有直接写 TimescaleDB 的路径；
- 设备 ACL 在列表、详情、历史、命令和实时通知中一致生效；
- 消费者崩溃、数据库故障和 Redis 重连不会静默丢数据；
- 单进程能够有序启动和优雅停止所有模块；
- 迁移和回滚流程已在旧数据副本上演练。
